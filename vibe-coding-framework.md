# 通用 Vibe Coding 框架

这是一套面向人类团队、主 agent 和 subagent 的 AI 协作开发框架。它适用于从想法、原型到可维护系统的开发过程，目标是在保持 vibe coding 速度的同时，控制业务语义漂移、上下文爆炸、跨域耦合、文档过期、测试不可复现和 subagent 冲突。

这套框架不是一个长清单，而是一组可协同工作的文档和工作流。核心原则是：先通过产品经理式聊天把需求问清楚，并把确认过的需求保存到本地；再形成设计真源、执行任务；先拆业务域，再拆技术任务；先定 contract，再写实现；每个任务只加载精准上下文；实现阶段必须按文档中的 module 顺序一次实现一个业务域或一个明确业务切片。每完成一个 module，都必须跑通该 module 的验收测试，测试通过后才允许进入下一个 module。

框架生成的项目文档默认使用中文，方便人类团队阅读、维护和评审。文件名、命令、API 名、schema 字段、代码标识符和外部协议名保持原样；正文说明、标题、任务卡、context pack、假设、风险、冲突报告和交付说明默认写中文，除非用户明确要求其他语言。

## 1. 框架目标

- 快速开发，但不牺牲业务语义、架构边界、测试和长期可维护性。
- 让人类、主 agent 和 subagent 能在同一套真源和边界下协作。
- 让需求澄清从聊天开始，但不只停留在聊天上下文里，而是落盘为可复用的需求真源。
- 让每个任务都有明确的读取范围、修改范围、验收入口和冲突处理方式。
- 让实现按 module 队列推进，避免 agent 跳过文档直接改代码。
- 让 agent 创建的文档默认可被中文团队直接阅读和维护。
- 避免 agent 为了局部测试把业务逻辑塞进编排层、胶水层或错误的模块。
- 避免默认把全部设计文档塞进上下文窗口。

## 2. 推荐文件体系

```text
AGENTS.md
docs/
  requirements/
    DISCOVERY_LOG.md
    PRODUCT_REQUIREMENTS.md
  design/
    PROJECT_DESIGN.md
    modules/
      <domain>.md
  ARCHITECTURE.md
  TASKS.md
  QUALITY.md
  DECISIONS.md
  context-packs/
    <task-or-domain>.md
```

### AGENTS.md

AI 执行纪律和红线。它说明 agent 应该怎么工作、什么不能做、冲突时何时暂停、subagent 如何交付。它不承载业务细节。

建议包含：真源优先级、开工读取规则、冲突处理规则、禁止跨域改动、禁止静默降级、禁止弱化测试、实现阶段一次只做一个业务域、交付报告格式。

必须明确：需求先行、文档第二、代码第三。缺少已确认需求、模块设计、架构映射、任务卡、context pack 或验收测试入口时，agent 不得开始改产品代码。

### docs/requirements/DISCOVERY_LOG.md

需求发现日志。它记录 agent 以产品经理身份和用户聊天时确认过的问题、答案、冲突、待确认点和关键取舍。它的作用是保存探索过程，避免重要上下文只留在对话窗口里。

建议包含：日期、访谈轮次、用户原话摘要、agent 理解、已确认需求、未确认问题、冲突或取舍、下一轮要问的问题。

它不应该成为最终设计文档，也不应该混入实现方案细节。已确认且稳定的内容应整理进 `PRODUCT_REQUIREMENTS.md`。

### docs/requirements/PRODUCT_REQUIREMENTS.md

产品需求真源。它承接需求发现阶段的确认结果，是后续 `PROJECT_DESIGN.md`、模块设计、任务卡和验收标准的输入。

建议包含：产品目标、用户和角色、核心场景、用户故事或任务流、功能需求、非功能需求、业务规则、边界和非目标、验收标准、开放问题、假设、需求变更记录。

在这个文件形成之前，agent 不应直接生成完整项目设计文档。设计文档可以引用它，但不要复制全部访谈过程。

### docs/design/PROJECT_DESIGN.md

项目总设计。它只放全局稳定内容，不塞所有模块细节。

建议包含：项目目标、非目标、用户和调用方、核心业务流程总览、核心概念词汇表、全局状态或生命周期概览、模块设计索引、全局错误语义、关键验收场景索引。

### docs/design/modules/*.md

按业务域拆分的模块设计文档，是业务域语义真源。每个模块文档应该让 subagent 在不理解整个系统的情况下，也能理解该业务域的职责、契约、规则、状态和测试。

模块文档按业务域拆，不按技术层拆。技术层映射放在 `ARCHITECTURE.md`。

### docs/ARCHITECTURE.md

架构边界和代码映射。它引用模块设计文档，不复制模块业务规则。

建议包含：分层结构、依赖方向、模块到代码目录映射、模块到设计文档映射、公共 contract 位置、编排层边界、跨域变更流程、架构红线。

### docs/TASKS.md

任务拆分、module 实现队列、任务卡和 subagent 分工。每个任务卡必须写清楚目标、业务域、module 顺序、context pack、允许修改、禁止修改、输入输出契约、验收测试、不做事项和主 agent 审核点。

`TASKS.md` 必须维护一个有序 module 队列。实现只能从队列中的当前 module 开始；当前 module 验收测试未通过时，不得开始下一个 module。

### docs/QUALITY.md

质量、测试、运行和 debug 守门。它定义单元测试、契约测试、集成测试、端到端脚本、真实依赖冒烟、fixture、mock、配置、错误处理、状态迁移、日志和合并前最低验收标准。

### docs/DECISIONS.md

关键决策日志。记录重要取舍、背景、采用方案、放弃方案、影响范围和复查条件。它用于防止后续 agent 重复讨论或反向实现旧方案。

### docs/context-packs/*.md

任务级精准上下文入口。context pack 只做索引，不复制大段设计内容。

建议包含：任务名称、业务域、必读文档与小节、相关 contract、相关测试、不需要读取的文档、本任务红线、已知风险、验收入口。

## 3. 需求发现与交互式成文流程

agent 不能一开始就直接写完整设计文档，也不能一次性脑补完整文档。它应该先扮演产品经理，和用户进行需求澄清聊天；每一轮只问 1 到 3 个高影响问题。需求澄清不是一次性问卷，而是循环访谈：用户每次回答后，agent 必须复述已确认内容，指出最重要的剩余缺口，再继续追问下一轮关键问题。当需求足够清晰时，先把确认结果落盘到 `docs/requirements/PRODUCT_REQUIREMENTS.md`，必要时把访谈过程追加到 `docs/requirements/DISCOVERY_LOG.md`，再进入设计文档阶段。

推荐流程：

1. 探索已有资料和代码，避免询问仓库里已经能回答的问题。
2. 以产品经理身份澄清项目目标、用户、调用方、核心场景和非目标。
3. 继续澄清业务规则、边界、角色权限、状态生命周期、错误语义、验收标准和高风险取舍。
4. 将每轮重要结论追加到 `DISCOVERY_LOG.md`，将已确认且稳定的需求整理到 `PRODUCT_REQUIREMENTS.md`。
5. 如果仍缺少目标、用户、主要流程、非目标、业务规则、验收标准或关键取舍，继续下一轮提问，不进入设计。
6. 当 agent 能清楚复述需求、指出开放问题，并认为需求足以支撑设计时，再开始写 `PROJECT_DESIGN.md`。
7. 识别业务域，确定每个域负责什么、不负责什么。
8. 为每个业务域编写模块设计文档。
9. 用 `ARCHITECTURE.md` 把模块映射到代码边界和依赖方向。
10. 用 `TASKS.md` 生成任务卡和 subagent 分工。
11. 为每个任务或业务域生成 context pack。
12. 用 `QUALITY.md` 固化测试、fixture、mock、验收和 debug 规则。

每次询问用户时，最多问 1 到 3 个高影响问题。可从仓库发现的问题先探索，不问用户。产品偏好、业务规则、公共契约和高风险取舍必须问用户。低影响实现细节可以采用推荐默认值，但要写入 `Assumptions`。

需求发现阶段结束前，agent 必须给出一段简短复述，说明：已确认需求、仍未确认但不阻塞的问题、下一步将写入哪些文档。用户明确同意，或需求文件中已经记录了可继续的假设后，才进入设计文档阶段。

### 需求发现完成门

进入设计文档前，至少要满足以下条件；不满足时继续追问，而不是只给计划：

- 产品目标、成功标准、目标用户和主要角色已经清楚。
- 核心场景、主要流程、重要异常流程和非目标已经清楚。
- 关键业务规则、权限、状态生命周期、错误语义和验收标准已经清楚。
- 外部依赖、公共契约、重要取舍和风险已经暴露。
- 剩余未知项已经被用户确认不阻塞，或写入 `PRODUCT_REQUIREMENTS.md` 的开放问题/假设。

## 4. 冲突处理机制

当用户当前要求和已有文档、契约或架构规则冲突时，agent 不能自行裁决，也不能静默覆盖文档。必须暂停并列出冲突项。

冲突报告应包含：

- 用户当前要求。
- 冲突文档和条款。
- 冲突类型：需求、架构边界、公共契约、测试质量、任务范围。
- 影响范围。
- 可选处理方式。
- 推荐选项。

用户必须选择：

1. 以新要求为准，更新相关文档和测试。
2. 以现有文档为准，调整当前请求。
3. 作为临时例外执行，记录风险和有效范围。

冲突级别建议：

- P0：公共契约、业务规则、数据安全、状态迁移冲突，必须用户选择。
- P1：模块边界、任务范围、测试策略冲突，默认暂停并建议选项。
- P2：命名、目录、说明文字冲突，agent 可推荐修正并说明。

## 5. 实现阶段规则

实现阶段是最容易漂移的阶段，必须强约束。

- 需求先行，文档第二，代码第三。缺少已确认需求或必要文档时，先补需求和文档，不改产品代码。
- 实现顺序由 `docs/TASKS.md` 中的 module 队列决定。
- 一次只实现一个业务域或一个明确业务切片。
- 没有 context pack 不开工。
- 开工前读取对应模块设计、架构映射、任务卡、context pack 和相关测试。
- 先补或确认测试，再实现。
- 实现后运行该域验收测试，必要时跑相关集成测试。
- 当前 module 的验收测试未通过，不得开始下一个 module。
- 没有测试或验收入口，不标记完成。
- 跨域修改必须升级为 contract change，由主 agent 审核。
- 用户要求与文档冲突时必须先让用户选择。

### 实现门禁

每个 module 开工前必须满足：

- `AGENTS.md` 已定义执行纪律和冲突处理规则。
- `docs/requirements/PRODUCT_REQUIREMENTS.md` 已记录当前 module 依赖的已确认需求、开放问题和假设。
- `PROJECT_DESIGN.md` 已列出项目目标、核心流程和 module 索引。
- 当前 module 已有 `docs/design/modules/<domain>.md`。
- `ARCHITECTURE.md` 已把当前 module 映射到代码边界。
- `TASKS.md` 已写明 module 队列和当前 module 任务卡。
- `QUALITY.md` 已定义当前 module 的验收测试或验证场景。
- 当前 module 已有 context pack。

不满足任一项时，agent 应暂停实现，先补齐需求和文档或向用户确认缺口。

### Module 验收门

每个 module 完成后必须：

- 运行该 module 的验收测试或验证场景。
- 修复失败项，且修复范围不得越过任务卡边界。
- 如果实现改变了业务语义、架构边界、contract 或验收方式，更新对应文档。
- 在交付报告中写明当前 module、测试结果、contract 变化、剩余风险和是否允许进入下一个 module。

## 6. 主 agent 与 subagent 协作

主 agent 是架构守门员，负责真源一致性、contract 合并、跨域冲突、集成测试和文档同步。subagent 只在任务卡声明的业务域或切片中工作。

subagent handoff 应包含：背景、当前业务域、必读 context pack、允许修改、禁止修改、预期测试、完成后报告格式。

subagent 完成后必须报告：改了什么、跑了哪些测试、剩余风险、是否改了 contract、是否需要主 agent 审核。

## 7. 防文档过期

- 改已确认需求先改 `docs/requirements/PRODUCT_REQUIREMENTS.md`。
- 改业务语义先改模块设计。
- 改代码边界先改架构映射。
- 改验收方式先改质量文档。
- 重大取舍写入决策日志。
- 每个任务结束时检查 contract 是否变化、模块职责是否仍匹配、测试是否覆盖任务卡验收。

## 8. 最低落地标准

一个项目至少应该具备：

- `AGENTS.md`：AI 执行纪律。
- `docs/requirements/PRODUCT_REQUIREMENTS.md`：已确认产品需求真源。
- `docs/requirements/DISCOVERY_LOG.md`：重要需求访谈和开放问题记录。
- `PROJECT_DESIGN.md`：项目总设计。
- 至少一份业务域模块设计文档。
- `ARCHITECTURE.md`：模块到代码边界映射。
- `TASKS.md`：当前里程碑任务卡。
- `QUALITY.md`：测试和验收规则。
- 当前任务的 context pack。

如果这些材料不完整，agent 应先帮助补齐文档或明确假设，再进入实现。
